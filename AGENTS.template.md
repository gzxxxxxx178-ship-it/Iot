# 通用项目协作规则模板

> 使用方法：复制本文件到目标项目根目录并重命名为 `AGENTS.md`，填写“项目配置”中的占位项，删除不适用的可选规则。除项目配置外，其余规则默认适用于所有软件项目。

## 1. 项目配置

```yaml
项目名称: {{PROJECT_NAME}}
项目目标: {{一句话说明项目解决的问题}}
仓库根目录: {{ABSOLUTE_OR_RELATIVE_PATH}}
主要技术栈:
  - {{LANGUAGE_OR_FRAMEWORK}}
目录结构:
  - {{PATH}}: {{PURPOSE}}
任务台账: {{TASK_FILE_OR_NONE}}
开发启动命令: {{COMMAND}}
格式检查命令: {{COMMAND_OR_NONE}}
静态检查命令: {{COMMAND_OR_NONE}}
单元测试命令: {{COMMAND}}
集成测试命令: {{COMMAND_OR_NONE}}
生产构建命令: {{COMMAND}}
提交语言: {{中文/英文}}
推送策略: {{默认不推送/允许推送}}
部署策略: {{默认不部署/允许部署到指定环境}}
```

项目配置缺失时，优先从仓库文件、脚本和 CI 配置中确认，不凭经验编造命令、路径或环境信息。

## 2. 指令优先级与任务边界

- 当前用户明确指令高于本模板中的默认工作偏好，但不得绕过安全、隐私和数据保护边界。
- 更深目录中的 `AGENTS.md` 可以补充该子目录规则；发生冲突时，以范围更具体的规则为准。
- 先判断任务属于咨询、审查、诊断、实现、部署还是监控，不把一种任务擅自扩展成另一种任务。
- 审查和诊断默认只读；除非用户要求修复，否则不得直接修改代码。
- 实现任务只处理实现目标所需范围，不顺带重构无关模块或增加未要求的功能。
- 部署、推送、删除数据、修改生产配置、发送外部消息等操作，必须有明确授权。
- 需求存在关键歧义且不同选择会明显改变结果时，一次只询问一个关键问题；边界清楚时直接执行。

## 3. 开始任务前

- 阅读项目入口文档、当前目录适用的规则、任务台账及相关模块，不无目的扫描全部大型目录。
- 执行 `git status --short`，识别用户已有修改、未跟踪文件和当前分支。
- 不覆盖、回滚、格式化或提交与当前任务无关的用户改动。
- 查明项目现有实现方式、命名约定、测试入口和构建方式，优先沿用现有结构。
- 对复杂任务先列出简短计划，明确修改范围、验证方式和风险点；简单任务可直接实施。
- 涉及数据库、外部接口、身份认证、硬件、生产环境或不可逆操作时，先确认备份和回滚路径。

## 4. 代码实现原则

### 4.1 优先级

代码质量按以下顺序判断：

1. 正确性与数据安全。
2. 可读性与可维护性。
3. 简洁性与一致性。
4. 性能与资源效率。
5. 技巧性和形式上的“高级”。

不得为了减少几行代码牺牲正确性，也不得为了展示设计模式引入没有现实收益的复杂度。

### 4.2 简洁性

- 采用满足当前需求的最简单完整方案，遵循 KISS 和 YAGNI。
- 删除重复逻辑前先确认语义确实相同；不要为了 DRY 把偶然相似的代码强行抽象。
- 函数和类应职责单一，名称应直接表达业务含义，避免 `data`、`temp`、`handleThing` 等模糊命名。
- 优先使用早返回减少深层嵌套；复杂条件应提取为有业务含义的变量或函数。
- 避免隐藏副作用、共享可变状态、魔法数字和散落的环境常量。
- 不保留废弃代码、注释掉的实现、临时调试输出、无用依赖和未使用导入。
- 不提前建设当前需求没有使用者的扩展点、通用框架或配置系统。

### 4.3 一致性

- 遵循仓库已有的目录、命名、错误处理、返回结构和依赖注入方式。
- 相同概念在数据库、后端、前端、协议和文档中使用一致的名称、单位和时间语义。
- 修改公共 API、数据结构或配置项时，检查全部生产者、消费者和兼容路径。
- 不在同一任务中进行无关的全仓格式化、重命名或目录迁移。

### 4.4 注释与文档字符串

- 注释解释“为什么这样做”、业务约束和非显然风险，不逐行翻译代码。
- 清晰的私有函数不强制添加重复性注释；公共 API、复杂算法、协议边界和兼容逻辑应有必要说明。
- TODO 必须包含具体原因和后续条件；不能用 TODO 代替当前任务必须完成的实现。
- 代码修改后同步清理已经失效的注释。

### 4.5 错误与边界处理

- 在系统边界校验输入，包括空值、范围、格式、权限、文件路径、外部响应和超时。
- 错误信息应能支持定位，但不得泄露密码、令牌、连接串、堆栈或内部实现细节。
- 不空捕获异常，不用固定成功响应掩盖失败，不把真实错误降级成无意义的 `null`。
- 明确重试次数、退避、超时和幂等性，避免无限重试和重复写入。
- 涉及并发、缓存、时间、金额、单位或浮点计算时，明确一致性和精度边界。

## 5. 变更范围与兼容性

- 追求“最小但完整”的变更：既不遗漏必要调用链，也不扩大到无关模块。
- 修复缺陷时优先定位根因，并增加能够复现该缺陷的回归测试。
- 公共接口、数据库 schema、配置格式、消息协议和文件格式默认保持向后兼容。
- 无法兼容时，必须提供迁移方案、版本说明和回滚方式。
- 数据库结构变更必须使用可审计的迁移文件，不依赖生产启动时自动修改 schema。
- 生成文件应通过源文件和生成工具更新，不直接手改无法持续维护的产物。
- 新增依赖前说明必要性，优先使用现有依赖或标准库，并检查许可证、维护状态和包体积影响。

## 6. 安全与隐私

- 密码、令牌、私钥、Cookie、数据库连接串和真实个人数据不得写入代码、Git、文档、测试夹具或日志。
- 敏感值通过环境变量、受控 Secret 或权限受限的本地文件注入，并提供无真实值的示例配置。
- 日志、截图、测试报告和终端输出必须脱敏。
- 认证只确认“是谁”，授权还必须确认“能否访问该资源”；所有用户数据和写操作都要检查资源归属。
- 默认采用最小权限、最小暴露面和最短合理有效期。
- 防范 SQL 注入、命令注入、路径穿越、XSS、CSRF、SSRF、不安全反序列化和开放重定向。
- 不因调试临时关闭生产安全控制；确需临时调整时必须限定环境、时间和访问范围。
- 如果发现凭据已经暴露，停止继续传播并建议轮换；不得把旧凭据复制到新位置。

## 7. 测试与验证

### 7.1 测试原则

- 测试行为和业务契约，不绑定无关实现细节。
- 根据风险选择验证强度：文档修改做格式和链接检查；局部逻辑做定向测试；跨模块、认证、数据或部署修改做完整回归。
- 至少考虑正常路径、边界值、失败路径、权限不足和历史缺陷回归。
- 测试不得连接生产数据库、生产消息代理、真实支付接口或不可控外部资源。
- 测试数据必须可重复、可隔离、可清理，不依赖执行顺序。
- 不为了让测试通过而删除断言、跳过失败测试或降低安全校验。

### 7.2 推荐验证顺序

1. 运行与改动最相关的定向测试。
2. 运行格式检查和静态检查。
3. 运行受影响模块的完整测试。
4. 运行生产构建或打包。
5. 高风险变更执行集成、端到端或回滚验证。

如果某项验证无法运行，必须说明具体原因、已完成的替代验证和剩余风险，不能写成“应该没问题”。

## 8. 强制自审门禁

实现完成后必须自行 review，未完成自审不得宣称任务完成。

### 第一遍：需求与正确性

- 每项用户要求是否都有对应实现和验证证据。
- 是否遗漏调用方、数据流、异常路径、权限或兼容场景。
- 是否解决根因，而不是只掩盖表面现象。
- 是否引入数据损坏、重复执行、竞态、资源泄漏或状态不同步。

### 第二遍：简洁性与可维护性

- 是否存在更短、更直接且同样可靠的实现。
- 是否出现重复逻辑、过度抽象、超长函数、模糊命名或隐藏副作用。
- 是否留下调试代码、无用依赖、失效注释或不必要的格式变化。
- 新代码是否符合项目现有结构和风格。

### 第三遍：安全与交付

- 是否泄露敏感信息或扩大权限、接口、CORS、网络和日志暴露面。
- 测试是否真实执行，结果是否与报告一致。
- 文档、配置示例、数据库迁移和任务状态是否同步。
- `git diff --check` 是否通过；完整 `git diff` 中是否只有本任务预期修改。

发现问题后先修复，再重新执行受影响的测试和自审步骤。

## 9. 文档与任务台账

- README、API 文档、配置示例、架构说明、迁移脚本和部署手册必须与代码保持一致。
- 项目存在任务台账时，以台账状态为事实来源；开始、受阻和完成时及时更新。
- 文档中的命令必须能够在声明的目录和环境中执行。
- 不编造版本、性能数据、测试结果、文献、链接或生产状态。
- 未确认的信息标记为“待确认”或“资料不足”，不要用确定语气填写占位内容。

## 10. Git 与文件操作

- 所有 Git 操作在项目配置声明的仓库根目录执行。
- 编辑前后检查 `git status`；提交前检查 `git diff --check`、`git diff --stat` 和完整差异。
- 一个提交只包含一个可独立说明和回滚的逻辑变更，提交信息说明结果而非过程。
- 完成并验证任务后创建本地提交；除非项目配置或用户明确允许，否则不推送。
- 不擅自 amend、rebase、force push、删除分支、覆盖历史或提交其他人的改动。
- 不使用 `git reset --hard`、`git clean -fd` 等破坏性命令处理未知改动。
- 不提交构建缓存、依赖目录、IDE 私有文件、临时文件或真实密钥。
- `.gitignore` 使用最小匹配范围，避免误伤应版本管理的文件。

## 11. 部署与生产变更

- 本地提交不等于获得部署授权；只有用户明确要求更新指定环境时才执行部署。
- 部署前确认目标环境、版本、配置、数据库迁移、备份、健康检查和回滚命令。
- 使用不可变版本或版本目录部署，不用来源不明的本地产物覆盖线上文件。
- 部署后至少验证进程、健康端点、关键依赖、核心接口和错误日志。
- 观察窗口内发现失败时立即停止扩大变更，并按预案回滚。
- 生产命令和报告不得回显密码、令牌、完整连接串或用户隐私数据。

## 12. 受阻与失败处理

- 命令失败时先读取完整错误和上下文，再定位根因；不机械重复同一操作。
- 外部服务不可用时，区分代码问题、环境问题、权限问题和网络问题。
- 缺少必要权限、凭据、硬件或用户决策时，完成所有安全的只读检查后再报告阻塞。
- 报告阻塞应包含：已确认事实、失败位置、已尝试措施、不能继续的原因和用户需要提供的一项信息。
- 不伪造成功，不以“部分完成”替代明确的剩余工作说明。

## 13. 完成定义

只有同时满足以下条件，任务才能标记为完成：

- 用户要求已完整实现，没有未经同意扩展范围。
- 相关测试、静态检查和构建通过，或已明确记录无法执行的原因与风险。
- 已完成三遍自审并修复发现的问题。
- 安全、兼容性、数据迁移和回滚风险已处理。
- 文档、配置示例和任务台账已同步。
- Git 差异只包含预期变更，没有覆盖用户已有工作。
- 已按项目规则创建本地提交，但未执行未经授权的推送或部署。

## 14. 最终交付格式

最终回复应简洁报告：

1. 完成了什么，以及用户可直接获得的结果。
2. 修改的关键文件。
3. 实际执行的测试、构建或线上验证结果。
4. 本地提交号，以及是否推送或部署。
5. 仍存在的风险、限制或唯一必要的下一步。

不得只描述过程，也不得用未经验证的判断代替结果。

## 15. 每次任务快速检查清单

```text
[ ] 已理解任务类型和范围
[ ] 已读取适用规则与相关代码
[ ] 已检查 git status 并保护用户改动
[ ] 实现采用最小、清晰、符合现有结构的方案
[ ] 已处理输入、错误、权限、安全和兼容边界
[ ] 已增加或更新必要测试
[ ] 已运行定向测试、静态检查和构建
[ ] 已检查 git diff --check、diff stat 和完整 diff
[ ] 已完成正确性、简洁性、安全性三遍自审
[ ] 已同步文档、迁移和任务台账
[ ] 已创建原子本地提交
[ ] 未执行未经授权的推送、部署或破坏性操作
```
