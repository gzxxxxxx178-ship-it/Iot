# 智慧农业 IoT 项目改造问题清单

## 使用规则

- 本文件是项目改造的唯一问题台账，用于记录范围、优先级、状态和验收标准。
- 默认只维护台账，不自动实施任务。用户明确指定任务编号后，一次只执行一个任务。
- 每项任务完成后必须经过验证，更新本文件状态，并按项目规范单独提交 Git。
- 不得把 Wi-Fi 密码、数据库密码、JWT 密钥、API Key、OAuth 密钥或支付密钥提交到 Git。

### P4-02 修复认证、会话与数据权限边界

- **状态**：`已完成`
- **问题**：CSRF 全局关闭且保留会话策略；聊天历史、支付订单、设备、规则和历史数据缺少完整的用户/租户归属校验；Redis 测试接口仍可任意读写键。
- **影响**：存在跨用户读取/删除数据、越权控制和 CSRF 风险；多用户场景不能安全上线。
- **范围**：Spring Security、OAuth2 会话、JWT、聊天、支付、设备、报警/自动化、历史数据和 Redis 控制器。
- **验收标准**：写操作具备明确的 CSRF 或无状态 JWT 防护；资源查询、修改和删除均绑定当前用户或租户；Redis 测试接口移除或严格限制；越权和未授权测试通过。

### P4-04 统一设备时间语义

- **状态**：`已完成`
- **问题**：ESP8266 上报的 `timestamp` 当前使用 `millis()`，表示设备运行时长而不是 Unix 时间。
- **影响**：历史数据、图表和 CSV 导出中的时间字段容易被误读，跨重启后也无法直接比较。
- **范围**：ESP8266 NTP 校时、上报载荷、后端 DTO/实体、历史查询和 CSV 导出。
- **验收标准**：上报时间使用 NTP/服务器校准的 Unix 时间，或字段明确命名为 `uptimeMillis`；后端保存和导出字段语义一致；重启和时区场景有验证。
- **完成结果**：ESP8266 载荷字段改为 `uptimeMillis`，后端新增同名持久化字段并保留旧 `timestamp` 输入兼容；历史排序、图表和展示统一使用服务端接收时间，CSV 明确导出设备运行时长（毫秒）和服务端接收时间，避免把运行时长误当作日期。

### P4-05 加强令牌存储并继续优化前端加载

- **范围**：登录回调、认证存储、Cookie/短期令牌策略、路由守卫和前端第三方依赖拆分。
- **验收标准**：令牌不再长期暴露在可被脚本读取的存储中；登录、刷新、退出和 401 处理完整；生产构建通过并记录主要 chunk 体积变化。
- **完成结果**：登录、注册和 Google OAuth 改为写入短期 HttpOnly JWT Cookie，前端不再从 `localStorage` 读取或发送令牌；axios 改为 `withCredentials` 并配合 XSRF Cookie，退出接口清除服务端Cookie，`/me` 用于启动恢复和401失效处理。针对 Cloudflare Pages 与 VPS 跨站部署中浏览器阻止第三方 Cookie 的情况，用户名登录和注册额外返回仅保存在当前页面内存中的 Bearer 令牌，不写入 Web Storage；401 会同步清理 Pinia 状态并返回登录页，路由切换不再重复请求 `/me`。前端按路由拆分页面与 ECharts/Element Plus 共享依赖，生产构建和 API 契约测试通过。

## P5：线上安全、可靠性与长期维护

### P5-01 轮换敏感凭据并加固 SSH

- **状态**：`待执行`

### P5-03 建立 TiDB 可审计数据库迁移基线

- **状态**：`待执行`
- **问题**：仓库生产配置为 Flyway 加 `ddl-auto=validate`，但 VPS 因旧版 Flyway 无法识别 TiDB 而临时关闭 Flyway，并恢复为 `ddl-auto=update`。

- **范围**：现有 TiDB schema 导出、迁移 baseline、版本化 SQL、测试数据库、生产配置、备份和迁移操作手册。
- **方法约束**：不得直接在线上切换为 `validate`；必须先导出现有结构，在隔离数据库验证 baseline 和全部迁移，再安排生产备份与维护窗口。Flyway 新版已提供独立 TiDB 支持模块，具体版本需要结合 Java/Spring Boot 兼容性验证：[Flyway TiDB 官方说明](https://documentation.red-gate.com/flyway/reference/database-driver-reference/tidb-titanium-db)。
- **验收标准**：新环境能够从空库重复建立结构；现有生产库能够安全 baseline；生产禁用 `ddl-auto=update` 并使用 `validate`；每次数据库变更均有版本、校验和、备份及失败处理记录。

### P5-04 完成同源认证与 Google OAuth 闭环

- **状态**：`待执行`
- **问题**：普通账号登录当前使用 HttpOnly Cookie 加页面内存 Bearer 兼容跨站限制，但内存令牌会随页面重载丢失；Google OAuth 仍主要依赖 Cloudflare Pages 与 VPS 之间的跨站 Cookie。
- **影响**：不同浏览器隐私策略下可能再次出现登录后 `/me` 返回 401、页面刷新掉线或 Google 登录失败。
- **范围**：Cloudflare Pages Functions 或自定义域名、同源 `/api` 代理、Cookie 属性、CORS、CSRF、OAuth 回调、短期授权码交换、登录恢复和退出流程。
- **方法约束**：不得把 JWT、OAuth access token 或 refresh token 放入 URL；授权码必须短期、一次性并绑定用户代理状态；保留最小 CORS 白名单和 CSRF 防护。Cloudflare Pages Functions 支持按路径执行代理逻辑：[Pages Functions 路由](https://developers.cloudflare.com/pages/functions/routing/)；Google 推荐由服务端接收并交换授权码：[OAuth Web Server 流程](https://developers.google.com/identity/protocols/oauth2/web-server)。
- **验收标准**：普通登录、注册、Google 登录、刷新页面、跨路由、退出和令牌过期均稳定；浏览器不依赖第三方 Cookie；Cookie 和令牌不暴露给非必要脚本；认证回归测试通过。
